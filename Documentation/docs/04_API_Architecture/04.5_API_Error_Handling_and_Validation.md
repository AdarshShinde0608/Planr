# Purpose

This document defines the complete error handling and validation strategy for the backend API.

Objectives:

* Consistent error responses
* Centralized exception handling
* Strong request validation
* Clear business rule enforcement
* Structured logging
* Better debugging
* Improved frontend developer experience

Every API endpoint must follow this specification.

---

# Error Handling Architecture

```text
Incoming Request

↓

Authentication

↓

Authorization

↓

Validation

↓

Controller

↓

Service

↓

Repository

↓

MongoDB / External APIs

↓

Exception

↓

Global Error Handler

↓

Standard Error Response
```

Errors must never be handled directly inside controllers unless recovery is required.

---

# Error Categories

The system recognizes six major categories of errors.

| Category              | Example                       |
| --------------------- | ----------------------------- |
| Validation Errors     | Missing required field        |
| Authentication Errors | Invalid Firebase token        |
| Authorization Errors  | Accessing another user's task |
| Business Rule Errors  | Deadline in the past          |
| Infrastructure Errors | MongoDB unavailable           |
| AI Service Errors     | Planner Agent timeout         |

Each category maps to a specific HTTP status code.

---

# Error Hierarchy

```text
ApplicationError

├── ValidationError
├── AuthenticationError
├── AuthorizationError
├── ResourceNotFoundError
├── ConflictError
├── BusinessRuleError
├── ExternalServiceError
├── AIProcessingError
├── DatabaseError
└── InternalServerError
```

Every custom exception must inherit from `ApplicationError`.

# Global Exception Handler

Every uncaught exception passes through a single middleware.

Responsibilities:

* Convert exceptions
* Hide internal implementation
* Log errors
* Attach correlation ID
* Return standard response

Controllers should never manually construct error responses.

---

# Validation Pipeline

```text
Request

↓

Zod Validation

↓

Business Validation

↓

Controller

↓

Service
```

Validation always occurs before business logic.

---

# Request Validation

All incoming requests must be validated using Zod.

Examples:

* Required fields
* String length
* Email format
* ISO date
* Enum validation
* Numeric limits

---

# Business Validation

Business validation occurs inside services.

Examples:

* Deadline must be in the future
* User owns task
* Schedule has available slots
* Habit already completed today
* Goal already completed

Business validation must not be placed inside Zod schemas.

# Database Error

Database errors should never expose raw MongoDB messages.

Convert:

```text
Duplicate Key Error

↓

EMAIL_ALREADY_EXISTS
```

Never expose stack traces.

---

# External Service Errors

External integrations include:

* OpenAI
* Google Calendar
* Firebase

Failures should be wrapped in domain-specific exceptions.

Example:

```text
Google Calendar Timeout

↓

CALENDAR_SYNC_FAILED
```

---

# Correlation IDs

Every request receives a unique correlation ID.

Example:

```http
X-Correlation-ID: 9baf2d2d...
```

Used for:

* Logs
* Worker tracing
* AI execution
* Monitoring

---

# Logging Strategy

Every error should log:

* Timestamp
* Correlation ID
* User ID
* Endpoint
* HTTP Method
* Error Code
* Stack Trace (server only)

Never log:

* Passwords
* Firebase tokens
* API keys
* Personal secrets

---

# AI Error Handling

If an AI agent fails:

```text
Planner Agent

↓

Retry

↓

Fallback

↓

User Notification
```

Failures should not crash the application.

---

# Retry Policy

Automatic retries:

| Component            | Retries |
| -------------------- | ------- |
| OpenAI               | 3       |
| Google Calendar      | 2       |
| Notification Service | 5       |
| Event Publishing     | 5       |

Use exponential backoff.

---

# Timeout Policy

Recommended timeouts:

| Component       | Timeout |
| --------------- | ------- |
| Planner Agent   | 15 sec  |
| Scheduler Agent | 20 sec  |
| Coach Agent     | 10 sec  |
| Reminder Agent  | 5 sec   |
| Google Calendar | 10 sec  |

Requests exceeding the timeout should fail gracefully.

# Best Practices

* Never return raw exceptions.
* Use meaningful error codes.
* Keep user-facing messages simple.
* Log technical details on the server only.
* Validate before executing business logic.
* Fail fast for invalid requests.
* Make AI failures recoverable.
* Use correlation IDs for traceability.

---

# Antigravity Development Instructions

When implementing error handling:

1. Create a global `ApplicationError` base class.
2. Implement specialized error classes for each category.
3. Add a centralized Express error middleware.
4. Validate every request with Zod before controllers.
5. Perform business validation inside services.
6. Convert infrastructure errors into domain-specific exceptions.
7. Include a correlation ID in every error response.
8. Log structured error information using a centralized logger.
9. Implement retry and timeout logic for AI agents and external services.
10. Write automated tests covering every error scenario.
