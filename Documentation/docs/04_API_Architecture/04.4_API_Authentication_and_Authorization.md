# Authentication Architecture

The application uses **Firebase Authentication** for identity management.

The backend **never stores passwords**.

```text id="auth-flow-001"
User

↓

Frontend

↓

Firebase Authentication

↓

Firebase ID Token

↓

Express Backend

↓

Firebase Admin SDK

↓

Verified User

↓

JWT Context

↓

Protected API
```

---

# Why Firebase Authentication?

Advantages:

* Secure password handling
* Google Sign-In support
* Email verification
* Password reset
* Multi-factor authentication (future)
* Trusted identity provider
* Reduced backend complexity

---

# Authentication Flow

## Step 1 — Login

User logs in using:

* Email & Password
* Google Sign-In

Firebase returns:

```typescript id="firebase-token"
Firebase ID Token
```

---

## Step 2 — API Request

The frontend includes the token:

```http id="auth-header"
Authorization: Bearer <Firebase_ID_Token>
```

---

## Step 3 — Backend Verification

Authentication middleware:

1. Extract token
2. Verify with Firebase Admin SDK
3. Decode user information
4. Retrieve internal user record
5. Attach user context to request

---

## Step 4 — Authorized Request

```text id="authorized-request"
Controller

↓

Service

↓

Repository
```

Business logic now has access to:

* User ID
* Email
* Role
* Permissions

---

# User Context

Every authenticated request contains:

```typescript id="request-user"
interface RequestUser {
  id: string;
  firebaseUid: string;
  email: string;
  role: UserRole;
}
```

Accessible through:

```typescript id="req-user"
req.user
```

---

# Session Management

The application is stateless.

Sessions are managed entirely through Firebase tokens.

Backend stores:

* No passwords
* No login sessions
* No authentication cookies

Benefits:

* Horizontal scalability
* Simpler deployments
* Reduced attack surface

# Authorization

Every protected endpoint requires:

* Valid Firebase token
* Existing user record
* Ownership validation

Authentication alone is not sufficient.

# Ownership Validation

Every resource belongs to a user.

Before any update:

```text id="ownership"
Task.userId

↓

Request.user.id

↓

Match?

↓

Allow

Else

403 Forbidden
```

This prevents unauthorized access to another user's data.

# Best Practices

* Never trust client input.
* Authenticate every protected request.
* Authorize every resource access.
* Validate ownership before updates.
* Use HTTPS exclusively.
* Keep authentication stateless.
* Log security events.
* Store secrets outside source code.

---

# Antigravity Development Instructions

When implementing authentication and authorization:

1. Use Firebase Authentication as the identity provider.
2. Verify Firebase ID tokens using the Firebase Admin SDK.
3. Store no passwords or authentication sessions in the backend.
4. Attach a strongly typed `req.user` object after authentication.
5. Implement ownership checks for every user-owned resource.
6. Enforce RBAC using middleware.
7. Protect all non-public endpoints.
8. Configure Helmet, CORS, and rate limiting globally.
9. Log all authentication and authorization events.
10. Write integration tests covering successful authentication, failed authentication, authorization failures, and ownership validation.
