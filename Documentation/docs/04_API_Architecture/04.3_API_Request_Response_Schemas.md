# Purpose

This document defines the complete request and response contracts for every REST API endpoint.

It serves as the foundation for:

* Frontend TypeScript types
* Backend DTOs
* Zod validation
* OpenAPI components
* AI agent integrations
* Mobile application APIs

Every endpoint must use these schemas.

---

# Design Principles

All schemas must:

* Be immutable
* Be versioned
* Be reusable
* Support OpenAPI generation
* Support Zod validation
* Share identical frontend/backend types

---

# Common Response Schema

Every successful API response follows the same structure.

```typescript
interface ApiSuccess<T> {
  success: true;
  message: string;
  data: T;
  meta?: PaginationMeta;
}
```

Example

```json
{
  "success": true,
  "message": "Task created successfully.",
  "data": {}
}
```

---

# Common Error Schema

```typescript
interface ApiError {
  success: false;
  error: {
    code: string;
    message: string;
    details?: object;
  };
}
```

Example

```json
{
  "success": false,
  "error": {
    "code": "TASK_NOT_FOUND",
    "message": "Task not found."
  }
}
```

# Validation Rules

Every request must be validated with Zod.

Example

```typescript
CreateTaskSchema

↓

title → required

deadline → ISO Date

priority → enum

description → max 1000 chars
```

Validation failures return:

```http
400 Bad Request
```

# Backward Compatibility

Rules:

* Never remove existing fields.
* Add new fields as optional first.
* Version breaking changes.
* Maintain API compatibility during migrations.

---

# Best Practices

* Reuse DTOs whenever possible.
* Keep request and response objects minimal.
* Never expose database internals.
* Return enums instead of arbitrary strings.
* Keep AI metadata in dedicated DTOs.
* Use ISO-8601 for all dates and times.

---

# Antigravity Development Instructions

When implementing API schemas:

1. Define all DTOs as shared TypeScript interfaces.
2. Generate Zod schemas from the same source where possible.
3. Ensure OpenAPI components reference these DTOs.
4. Keep frontend and backend types synchronized.
5. Validate every request before reaching controllers.
6. Never expose MongoDB-specific fields directly.
7. Use enums consistently across all layers.
8. Ensure response objects always follow the common success/error format.
9. Support automatic code generation for frontend clients.
10. Write schema validation tests for every DTO.
