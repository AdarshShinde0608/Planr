# Purpose

This document defines the API architecture for **The Last-Minute Life Saver**.

The API layer serves as the communication bridge between:

* Frontend
* Backend Services
* AI Agents
* Background Workers
* External Services (Google Calendar, Firebase, OpenAI)

The architecture is designed to be:

* RESTful
* Versioned
* Secure
* Stateless
* Event-driven
* AI-friendly
* Easily extensible

---

# API Design Principles

Every API must follow these principles:

* Stateless
* Resource-oriented
* Predictable URLs
* Consistent responses
* Versioned
* Secure by default
* Idempotent where applicable

---

# API Base URL

Development

```text
http://localhost:5000/api/v1
```

Production

```text
https://api.lastminutelifesaver.com/api/v1
```

All APIs must be versioned.

---

# API Versioning

Use URI versioning.

Example:

```text
/api/v1/tasks

/api/v1/schedules

/api/v1/goals
```

Future versions:

```text
/api/v2
```

Older versions remain operational until officially deprecated.

# HTTP Methods

| Method | Purpose          |
| ------ | ---------------- |
| GET    | Retrieve data    |
| POST   | Create resource  |
| PUT    | Replace resource |
| PATCH  | Partial update   |
| DELETE | Soft delete      |

Use methods according to REST semantics.

---

# Standard Response Format

## Success

```json
{
  "success": true,
  "data": {},
  "meta": {},
  "message": "Success"
}
```

---

## Error

```json
{
  "success": false,
  "error": {
    "code": "TASK_NOT_FOUND",
    "message": "Task not found."
  }
}
```

Every endpoint must follow this structure.

---

# HTTP Status Codes

| Code | Meaning                 |
| ---- | ----------------------- |
| 200  | Success                 |
| 201  | Created                 |
| 204  | No Content              |
| 400  | Validation Error        |
| 401  | Unauthorized            |
| 403  | Forbidden               |
| 404  | Not Found               |
| 409  | Conflict                |
| 422  | Business Rule Violation |
| 429  | Too Many Requests       |
| 500  | Internal Server Error   |

Avoid using `200` for failed operations.

# Authentication

Authentication uses Firebase Authentication.

Flow:

```text
Frontend

↓

Firebase Login

↓

Firebase ID Token

↓

Authorization Header

↓

Backend Verification

↓

User Context
```

Backend never stores passwords.

---

# Authorization

Every authenticated request contains:

```http
Authorization: Bearer <Firebase_ID_Token>
```

Middleware:

1. Verify Firebase token.
2. Resolve internal user.
3. Attach user context.
4. Continue request.

---

# Validation

All request payloads are validated using Zod.

Validation includes:

* Required fields
* Data types
* String lengths
* Date formats
* Enum values
* Business constraints

Requests failing validation return **400 Bad Request**.

---

# Pagination

Large datasets must support pagination.

Example:

```http
GET /tasks?page=1&limit=20
```

Response:

```json
{
  "data": [],
  "meta": {
    "page": 1,
    "limit": 20,
    "total": 143,
    "totalPages": 8
  }
}
```

---

# Filtering

Example:

```http
GET /tasks?status=Pending&priority=High
```

Supported filters vary by module.

---

# Sorting

Example:

```http
GET /tasks?sort=deadline&order=asc
```

Supported:

* asc
* desc

---

# Searching

Example:

```http
GET /tasks?search=dbms
```

Search should be case-insensitive.

---

# Idempotency

Endpoints like task creation may accept an optional idempotency key.

Header:

```http
Idempotency-Key: uuid
```

Duplicate requests should not create duplicate resources.

---

# Rate Limiting

Protect public endpoints.

Suggested limits:

| Endpoint     | Limit   |
| ------------ | ------- |
| Login        | 10/min  |
| Register     | 5/min   |
| AI Assistant | 30/min  |
| Task APIs    | 120/min |

Rate limiting should be configurable.

---

# Error Handling

All exceptions pass through a global error handler.

Responsibilities:

* Convert errors to API responses
* Log failures
* Hide internal details
* Preserve correlation IDs

---

# Correlation IDs

Every request receives a unique ID.

Example:

```http
X-Correlation-ID: 3f5b9d2f...
```

Used for:

* Logs
* Events
* Worker tracing
* AI execution tracking

---

# API Documentation

Use OpenAPI 3.1.

Generate:

* Interactive Swagger UI
* JSON specification
* TypeScript client SDK

Documentation should be updated alongside implementation.

---

# Security

* HTTPS only
* CORS configuration
* Helmet security headers
* Input sanitization
* JWT verification
* Rate limiting
* CSRF protection where applicable

Sensitive information must never appear in responses.

---

# Caching

Cache suitable GET endpoints.

Examples:

* Dashboard summary
* Daily schedule
* User preferences
* Analytics snapshot

Invalidate cache on relevant updates.

---

# API Logging

Log:

* Method
* Route
* User ID
* Correlation ID
* Response time
* Status code

Do not log sensitive payloads.

# Best Practices

* Keep controllers thin.
* Return consistent response structures.
* Validate every request.
* Use descriptive error codes.
* Publish domain events after successful writes.
* Avoid business logic in routes.
* Version APIs from day one.
* Document every endpoint.

---

# Antigravity Development Instructions

When implementing the API layer:

1. Create a versioned `/api/v1` structure.
2. Implement feature-based route modules.
3. Use Zod for all request validation.
4. Verify Firebase tokens through middleware.
5. Return standardized success and error responses.
6. Generate correlation IDs for every request.
7. Publish domain events after successful persistence.
8. Support pagination, filtering, searching, and sorting.
9. Generate OpenAPI documentation automatically.
10. Keep controllers minimal and delegate business logic to services.
