# Purpose

This document defines the documentation strategy for **The Last-Minute Life Saver**.

Documentation is treated as a core project deliverable rather than an afterthought. Every feature, service, API, workflow, and operational procedure must be documented to ensure long-term maintainability, efficient onboarding, and reliable production support.

---

# Objectives

The documentation strategy should:

* Maintain a single source of truth
* Reduce onboarding time
* Improve maintainability
* Standardize documentation quality
* Support future contributors
* Preserve architectural decisions
* Simplify production operations

---

# Documentation Principles

Documentation should be:

* Accurate
* Version controlled
* Easy to discover
* Easy to update
* Consistent
* Reviewable
* Developer-friendly

Documentation should evolve alongside the codebase.

# Best Practices

* Keep documentation close to the code.
* Write documentation during development, not afterward.
* Prefer diagrams for complex workflows.
* Keep examples up to date.
* Archive deprecated documentation instead of deleting it.
* Review documentation during every feature review.
* Maintain a consistent writing style across all documents.
* Validate documentation links regularly.

---

# Antigravity Development Instructions

When implementing the documentation strategy:

1. Treat documentation as a required deliverable for every feature and architectural change.
2. Store all documentation in the `/docs` directory using the standardized folder hierarchy.
3. Use Markdown for all technical documents and Mermaid for architecture diagrams where applicable.
4. Ensure every document follows the standard template with purpose, scope, dependencies, and revision history.
5. Update documentation whenever code, APIs, infrastructure, AI prompts, or operational procedures change.
6. Require documentation review as part of every Pull Request.
7. Maintain searchable, cross-linked documentation with consistent naming conventions.
8. Generate API documentation automatically from OpenAPI specifications where possible.
9. Maintain a living knowledge base for recurring operational and development issues.
10. Continuously audit documentation for accuracy, completeness, and outdated content.
