# Purpose

This runbook defines the operational procedures required to deploy, monitor, maintain, troubleshoot, and recover **The Last-Minute Life Saver** in production.

It serves as the primary reference for DevOps engineers, SREs, production support teams, and incident responders.

---

# Objectives

The runbook should enable teams to:

* Deploy safely
* Monitor platform health
* Respond to incidents
* Recover failed services
* Perform routine maintenance
* Protect production data
* Restore services quickly

# Service Restart Procedure

Restart order:

1. Workers
2. Backend
3. AI Orchestrator
4. Frontend

Avoid restarting the database unless necessary.

---

# AI Provider Failure

If the primary AI provider fails:

```text id="ops005"
Primary Provider

↓

Retry

↓

Fallback Provider

↓

Cached Response (if available)

↓

Graceful Degradation
```

Log all failover events.

---

# Queue Management

Monitor:

* Pending jobs
* Failed jobs
* Retry count
* Dead Letter Queue

Actions:

* Retry transient failures
* Investigate recurring failures
* Purge invalid jobs only after review

---

# Database Maintenance

Routine tasks:

* Backup verification
* Index maintenance
* Slow query review
* Vacuum/optimization
* Storage growth monitoring

Avoid schema changes during peak traffic.

---

# Redis Maintenance

Verify:

* Memory utilization
* Eviction policy
* Persistence
* Replication health
* Cache hit ratio

Restart only during maintenance windows when possible.

---

# Scheduled Maintenance

Before maintenance:

* Notify stakeholders
* Verify backups
* Freeze deployments
* Prepare rollback plan

After maintenance:

* Validate health checks
* Confirm application functionality
* Close maintenance window

---

# Backup Verification

Daily:

* Database backup
* Configuration backup
* Object storage metadata
* Secrets backup (encrypted)

Verify backup integrity through periodic restoration tests.

# Rollback Procedure

Rollback if:

* Critical regression detected
* Security issue introduced
* Performance degradation exceeds thresholds
* Deployment validation fails

Rollback steps:

1. Deploy previous release
2. Restore compatible configuration
3. Validate health
4. Monitor closely

---

# Security Events

* Unauthorized access
* Credential leaks
* Suspicious AI usage
* Excessive failed logins
* Privilege escalation attempts

Escalate according to the incident severity policy.

# Operational Best Practices

* Automate repetitive operational tasks.
* Keep runbooks updated after every incident.
* Test backups regularly.
* Practice disaster recovery exercises.
* Monitor trends rather than isolated events.
* Validate deployments before announcing completion.
* Keep production access restricted and audited.
* Document every operational change.

---

# Antigravity Development Instructions

When operating **The Last-Minute Life Saver** in production:

1. Follow standardized startup, shutdown, deployment, rollback, and recovery procedures.
2. Monitor infrastructure, APIs, AI services, databases, queues, and notifications continuously.
3. Respond to incidents according to defined severity levels and escalation procedures.
4. Validate every deployment using automated health checks and production smoke tests.
5. Maintain verified backups and perform scheduled restoration drills.
6. Use AI provider failover mechanisms to preserve core functionality during outages.
7. Record post-incident reviews for all major operational events and track corrective actions.
8. Continuously review capacity metrics to support proactive scaling.
9. Restrict production access using least-privilege principles and maintain complete audit logs.
10. Treat the runbook as a living operational document and update it after every significant infrastructure or process change.
