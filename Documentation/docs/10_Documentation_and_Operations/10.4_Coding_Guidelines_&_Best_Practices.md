# Purpose

This document defines the official engineering standards for **The Last-Minute Life Saver**.

It ensures consistency, readability, maintainability, scalability, and security across the entire codebase.

Every contributor is expected to follow these guidelines.

---

# Objectives

The coding standards should:

* Improve code quality
* Reduce technical debt
* Increase maintainability
* Encourage reusable components
* Minimize defects
* Improve team collaboration

---

# Engineering Principles

Every feature should follow:

* Clean Code
* SOLID
* DRY (Don't Repeat Yourself)
* KISS (Keep It Simple)
* YAGNI (You Aren't Gonna Need It)
* Separation of Concerns
* Single Responsibility

---

# General Coding Rules

Code should be:

* Readable
* Predictable
* Testable
* Modular
* Reusable
* Documented
* Consistent

Code is written for humans first, computers second.

---

# Naming Conventions

## Variables

Use descriptive camelCase.

Good:

```typescript id="code001"
const completedTaskCount = 12;
```

Avoid:

```typescript id="code002"
const c = 12;
```

---

## Functions

Function names should describe actions.

Good:

```typescript id="code003"
calculateSchedule()
createReminder()
generateDailyPlan()
```

---

## Classes

Use PascalCase.

```typescript id="code004"
class PlannerAgent {}

class TaskService {}

class NotificationWorker {}
```

---

## Interfaces

Prefix with **I** only if required by the project's convention. Otherwise, prefer descriptive names.

```typescript id="code005"
interface UserProfile {}

interface ScheduleOptions {}
```

---

## Constants

Use UPPER_SNAKE_CASE.

```typescript id="code006"
const MAX_TASKS_PER_DAY = 25;
```

---

# File Naming

Use:

```text id="code007"
task.service.ts

planner.agent.ts

notification.worker.ts

dashboard.controller.ts
```

Rules:

* lowercase
* kebab-style filenames
* descriptive names

---

# Folder Organization

```text id="code008"
feature/

controllers/

services/

repositories/

validators/

types/

utils/

tests/
```

Each folder should have a single responsibility.

---

# Function Design

Functions should:

* Perform one task
* Be short
* Have descriptive names
* Avoid side effects
* Return predictable results

Target length:

**Less than 40 lines**

---

# Class Design

Classes should:

* Follow Single Responsibility Principle
* Use dependency injection
* Avoid unnecessary state
* Be easy to test

---

# TypeScript Guidelines

Always:

* Enable strict mode
* Avoid `any`
* Prefer explicit return types
* Use readonly where appropriate
* Leverage discriminated unions
* Prefer enums only when necessary

Example:

```typescript id="code009"
function calculateDuration(task: Task): number {
  return task.duration;
}
```

---

# React Best Practices

Prefer:

* Functional components
* Hooks
* Composition over inheritance
* Custom hooks for reusable logic
* Lazy loading where appropriate

Avoid:

* Deep prop drilling
* Large components
* Unnecessary re-renders

---

# State Management

Use:

* Local state for component-specific data
* Global state only for shared application state
* Server state libraries for API data
* Memoization where beneficial

Avoid storing derived state.

---

# Backend Best Practices

Controllers should:

* Validate requests
* Delegate business logic
* Return responses

Business logic belongs in services.

Repositories should only access the database.

---

# API Design

Every endpoint should:

* Validate input
* Return consistent responses
* Handle errors gracefully
* Be idempotent where appropriate

---

# Database Guidelines

Prefer:

* Indexed queries
* Transactions for critical operations
* Parameterized queries
* Migrations instead of manual schema changes

Avoid:

* N+1 queries
* Unbounded table scans
* Raw SQL without justification

---

# AI Module Standards

Each AI Agent should:

* Have a single responsibility
* Use structured prompts
* Validate outputs
* Log execution metrics
* Handle provider failures gracefully

Prompt templates should be version controlled.

---

# Error Handling

Never ignore exceptions.

Preferred pattern:

```typescript id="code010"
try {
  await service.execute();
} catch (error) {
  logger.error(error);
  throw new ApplicationError();
}
```

Expose user-friendly messages while logging detailed diagnostics internally.

---

# Logging Guidelines

Log:

* Errors
* Warnings
* Important business events
* Performance metrics

Never log:

* Passwords
* Tokens
* API keys
* Personal user information

Use structured JSON logging.

---

# Security Practices

Always:

* Validate input
* Sanitize output
* Use parameterized queries
* Escape user-generated content
* Apply least privilege
* Store secrets securely

Never:

* Trust client input
* Hardcode credentials
* Disable security checks

---

# Performance Guidelines

Optimize:

* Database queries
* Bundle size
* API payloads
* Rendering
* Caching
* Background processing

Measure before optimizing.

---

# Testing Requirements

Every feature should include:

* Unit tests
* Integration tests
* AI evaluation (if applicable)

Critical workflows require E2E coverage.

---

# Documentation Standards

Document:

* Public APIs
* Complex algorithms
* Architectural decisions
* Configuration
* Non-obvious business rules

Avoid redundant comments that repeat the code.

---

# Dependency Management

Only introduce new dependencies if:

* They solve a real problem
* They are actively maintained
* They have acceptable licensing
* They pass security review

Remove unused dependencies promptly.

---

# Code Review Checklist

Reviewers should verify:

* Correctness
* Readability
* Performance
* Security
* Test coverage
* Documentation updates
* Naming consistency
* Architectural alignment

---

# Refactoring Guidelines

Refactor when:

* Code duplication exists
* Complexity increases
* Readability declines
* Performance bottlenecks appear

Refactoring should not change observable behavior.

---

# Technical Debt

Track technical debt explicitly.

Every item should include:

* Description
* Impact
* Priority
* Owner
* Planned resolution

Avoid accumulating undocumented debt.

---

# Anti-Patterns

Avoid:

* God classes
* Massive functions
* Circular dependencies
* Magic numbers
* Global mutable state
* Duplicate logic
* Deep nesting
* Premature optimization

# Best Practices

* Write self-documenting code.
* Prefer composition over inheritance.
* Keep functions focused.
* Minimize shared mutable state.
* Review code before requesting review.
* Optimize for readability.
* Remove dead code promptly.
* Continuously improve the codebase through incremental refactoring.

---

# Antigravity Development Instructions

When writing code for **The Last-Minute Life Saver**:

1. Follow Clean Code, SOLID, DRY, KISS, and YAGNI principles consistently.
2. Keep business logic in services, controllers thin, and repositories responsible only for data access.
3. Use TypeScript strict mode and avoid the `any` type unless there is a documented exception.
4. Validate all external input and sanitize all user-generated output.
5. Write automated tests for every new feature and maintain the project's coverage targets.
6. Keep functions small, descriptive, and focused on a single responsibility.
7. Use structured logging while protecting sensitive information.
8. Update documentation whenever architectural or behavioral changes are introduced.
9. Treat code reviews as collaborative quality improvements rather than approval checkpoints.
10. Continuously reduce technical debt through planned refactoring and adherence to engineering standards.
