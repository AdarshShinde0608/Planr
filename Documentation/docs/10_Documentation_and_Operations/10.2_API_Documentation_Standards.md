# Purpose

This document defines the standards for documenting all APIs in **The Last-Minute Life Saver**.

Every REST endpoint, AI endpoint, webhook, and internal service interface must be documented consistently to ensure maintainability, discoverability, and ease of integration.

---

# Objectives

API documentation should:

* Be accurate
* Be automatically generated where possible
* Stay synchronized with implementation
* Be easy to understand
* Support frontend, backend, and third-party developers
* Reduce integration errors

---

# Documentation Principles

API documentation should be:

* Complete
* Version-controlled
* Human-readable
* Machine-readable
* Searchable
* Consistently formatted

Documentation is part of the API—not an optional addition.

---

# API Documentation Structure

Each endpoint must document:

* Purpose
* URL
* HTTP Method
* Authentication
* Permissions
* Path Parameters
* Query Parameters
* Headers
* Request Body
* Response Body
* Error Responses
* Rate Limits
* Examples
* Notes

# Endpoint Template

Each endpoint should follow this structure.

## Endpoint

```text id="api002"
POST /api/v1/tasks
```

### Description

Creates a new task.

### Authentication

Required

Bearer JWT Token

### Permissions

Authenticated User

### Request

```json id="api003"
{
  "title": "Complete DBMS Assignment",
  "deadline": "2026-07-10T18:00:00Z",
  "priority": "HIGH"
}
```

### Success Response

```json id="api004"
{
  "id": "task_123",
  "status": "CREATED"
}
```

### Error Responses

* 400 Bad Request
* 401 Unauthorized
* 403 Forbidden
* 409 Conflict
* 500 Internal Server Error

# URL Standards

Use:

```text id="api005"
/api/v1/tasks

/api/v1/goals

/api/v1/schedules
```

Avoid verbs in URLs.

Preferred:

```text id="api006"
POST /tasks
```

Avoid:

```text id="api007"
POST /createTask
```

Use URI versioning.

Example

```text id="api017"

↓
```

Rules:

* No breaking changes within a major version
* Deprecate before removal
* Maintain migration guides

# Best Practices

* Generate documentation from source code where possible.
* Keep request and response examples realistic.
* Use consistent error formats.
* Version APIs explicitly.
* Include authentication requirements on every protected endpoint.
* Document deprecations early.
* Review documentation during API code reviews.
* Keep examples synchronized with implementation.

---

# Antigravity Development Instructions

When implementing API documentation:

1. Adopt **OpenAPI 3.1** as the canonical API specification.
2. Generate interactive documentation automatically from backend annotations and schemas.
3. Document every endpoint with requests, responses, authentication, validation rules, examples, and error codes.
4. Use standardized response and error formats across the entire API.
5. Version all APIs using URI versioning (`/api/v1`).
6. Publish searchable interactive documentation for developers.
7. Maintain changelogs and migration guides for every API version.
8. Validate OpenAPI specifications during CI/CD to prevent documentation drift.
9. Generate SDKs from the OpenAPI specification where applicable.
10. Treat undocumented or outdated endpoints as release-blocking documentation defects.
