# Purpose

This document defines the architecture, responsibilities, lifecycle, and implementation strategy for all background workers.

Background workers execute long-running and asynchronous operations outside the request-response lifecycle.

Their responsibilities include:

* AI Processing
* Schedule Generation
* Reminder Delivery
* Calendar Synchronization
* Analytics Processing
* Learning Engine Updates
* Notification Dispatch
* Cleanup Tasks

---

# Why Background Workers?

Without workers:

```text
User Request

↓

AI Processing

↓

Schedule Generation

↓

Analytics

↓

Notifications

↓

Response (15–30 seconds)
```

With workers:

```text
User Request

↓

Save Data

↓

Publish Event

↓

Return Response (<500ms)

↓

Background Workers Process Tasks
```

This provides a faster and more responsive user experience.

---

# Worker Architecture

```text
REST API

↓

Event Bus

↓

BullMQ Queue

↓

Worker

↓

Business Logic

↓

Database

↓

Optional New Event
```

Workers should be stateless and independently scalable.

# Queue Configuration

Each worker has its own BullMQ queue.

```text
planner-analysis

scheduler-generation

calendar-sync

reminder

analytics

learning

coach

notification

cleanup
```

Queues should be isolated to avoid resource contention.

---

# Worker Lifecycle

```text
Receive Job

↓

Validate Payload

↓

Load Context

↓

Execute Logic

↓

Update Database

↓

Publish Event

↓

Complete Job
```

If execution fails:

* Retry
* Log error
* Move to Dead Letter Queue after max attempts

---

# Retry Strategy

| Attempt |      Delay |
| ------- | ---------: |
| 1       |  Immediate |
| 2       |  5 seconds |
| 3       | 30 seconds |
| 4       |  2 minutes |
| 5       | 10 minutes |

After five failures, move the job to the DLQ.

---

# Resource Management

Workers should:

* Process jobs concurrently where safe
* Respect rate limits for external APIs
* Avoid memory leaks
* Release resources after each job
* Limit AI concurrency to control costs

---

# Monitoring

Track:

* Queue length
* Average processing time
* Job success rate
* Retry count
* Worker uptime
* AI latency
* Notification success
* Calendar sync success

Expose metrics for future monitoring dashboards.

---

# Failure Recovery

If a worker crashes:

1. BullMQ re-queues unfinished jobs.
2. Worker restarts automatically.
3. Jobs resume processing.
4. Alerts are generated if failures exceed thresholds.

No completed job should be processed twice.

---

# Scalability

Workers should scale horizontally.

Example:

```text
Planner Queue

↓

Planner Worker 1

Planner Worker 2

Planner Worker 3
```

Additional workers can be added without changing application logic.

---

# Best Practices

* Keep workers stateless.
* Make all jobs idempotent.
* Avoid direct worker-to-worker calls.
* Publish events instead of chaining logic.
* Log execution times.
* Use structured logging.
* Validate job payloads before execution.
* Keep AI prompts external to worker code.

# Worker Directory Structure

```text
workers/

├── planner.worker.ts
├── scheduler.worker.ts
├── reminder.worker.ts
├── calendar.worker.ts
├── analytics.worker.ts
├── learning.worker.ts
├── coach.worker.ts
├── notification.worker.ts
├── cleanup.worker.ts
├── base.worker.ts
└── worker.registry.ts
```

`base.worker.ts` should provide common functionality such as logging, retries, validation, and metrics. `worker.registry.ts` should register all workers during application startup.

---

# Antigravity Development Instructions

When implementing background workers:

1. Create a common `BaseWorker` abstraction.
2. Register each worker independently.
3. Connect workers to BullMQ queues.
4. Validate payloads before processing.
5. Publish domain events after successful execution.
6. Implement retry and dead-letter handling.
7. Record metrics and execution times.
8. Ensure workers are stateless and horizontally scalable.
9. Separate AI processing from business logic.
10. Design workers to support future distributed deployment.
