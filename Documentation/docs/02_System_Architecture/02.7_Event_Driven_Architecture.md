# Purpose

This document defines the Event-Driven Architecture (EDA) used throughout the application.

The event system enables:

* Loose coupling between services
* Asynchronous execution
* Background processing
* AI orchestration
* Scalability
* Reliability
* Fault isolation

Instead of directly calling services, components communicate by publishing and subscribing to events.

---

# Why Event-Driven Architecture?

Traditional applications often rely on synchronous service-to-service calls.

Example:

```text
Task Service
    ↓
Scheduler Service
    ↓
Reminder Service
    ↓
Analytics Service
```

This creates tight coupling.

Instead, our application follows:

```text
Task Service
      │
Publish TaskCreated
      │
      ▼
──────────── Event Bus ────────────
      │
      ├──────── Scheduler Worker
      ├──────── Reminder Worker
      ├──────── Analytics Worker
      └──────── Learning Worker
```

No service needs to know who is consuming its events.

# Event Bus

The application contains a centralized internal Event Bus.

Responsibilities:

* Register publishers
* Register subscribers
* Dispatch events
* Queue jobs
* Retry failed jobs
* Dead-letter handling
* Event logging

---

# Event Structure

Every event must follow a consistent schema.

```typescript
interface DomainEvent<T> {
  eventId: string;
  eventType: string;
  version: number;
  timestamp: string;
  source: string;
  correlationId: string;
  userId: string;
  payload: T;
}
```

---

# Event Naming Convention

Use the format:

```
<Entity><Action>
```

Examples:

```
TaskCreated
TaskUpdated
TaskCompleted
GoalCreated
HabitCompleted
ScheduleGenerated
ReminderSent
CalendarSynced
```

Past-tense names indicate completed actions.

# Retry Policy

Failed jobs should follow exponential backoff.

| Attempt |      Delay |
| ------- | ---------: |
| 1       |  Immediate |
| 2       |  5 seconds |
| 3       | 30 seconds |
| 4       |  2 minutes |
| 5       | 10 minutes |

After the final attempt, move the job to the Dead Letter Queue.

---

# Dead Letter Queue (DLQ)

Jobs that repeatedly fail are moved to a dedicated DLQ.

Reasons include:

* AI response validation failure
* Database outage
* Third-party API timeout
* Unexpected exceptions

Administrators can inspect and replay DLQ jobs.

---

# Event Ordering

Some events depend on previous ones.

Correct sequence:

```text
TaskCreated

↓

PlannerAnalysisCompleted

↓

ScheduleGenerated

↓

ReminderQueued

↓

ReminderSent
```

Workers must not process dependent events out of order.

---

# Idempotency

Workers must safely handle duplicate events.

Examples:

* Sending the same reminder twice should be prevented.
* Completing an already completed task should not affect analytics.
* Calendar synchronization should avoid duplicate events.

Idempotency keys should be stored where appropriate.

---

# Event Versioning

Every event includes a version number.

Example:

```json
{
  "eventType": "TaskCreated",
  "version": 1
}
```

Future payload changes should increment the version while maintaining backward compatibility.

---

# Correlation IDs

Every request generates a Correlation ID.

Example flow:

```text
HTTP Request

↓

Correlation ID Generated

↓

TaskCreated

↓

ScheduleGenerated

↓

ReminderQueued

↓

AnalyticsUpdated
```

This ID links logs across services for easier debugging.

* Failed events
* Retry counts

Metrics should be exposed for dashboards (e.g., Prometheus/Grafana in future).

---

# Security

* Validate all event payloads.
* Do not include secrets in events.
* Encrypt sensitive data at rest.
* Restrict internal event publishing to trusted services.
* Audit critical domain events.

---

# Scalability

Future upgrades:

* RabbitMQ
* Apache Kafka
* Redis Streams
* Distributed worker clusters
* Multi-region event processing

The event model should remain unchanged when replacing the underlying messaging technology.

---

# Best Practices

* Publish events only after successful database commits.
* Keep payloads focused and lightweight.
* Avoid circular event chains.
* Ensure handlers are idempotent.
* Separate command handling from event handling.
* Prefer asynchronous processing for long-running tasks.

---

# Antigravity Development Instructions

When implementing the event-driven system:

1. Create a centralized Event Bus abstraction.
2. Use BullMQ with Redis as the initial messaging layer.
3. Publish events only after successful persistence.
4. Register workers independently.
5. Implement retry and DLQ support.
6. Include correlation IDs in all events.
7. Version every event schema.
8. Validate event payloads before processing.
9. Ensure all workers are idempotent.
10. Design the event layer so that Redis can later be replaced with Kafka or RabbitMQ without changing business logic.
