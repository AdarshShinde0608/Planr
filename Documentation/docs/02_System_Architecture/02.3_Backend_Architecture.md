# Purpose

This document defines the backend architecture of **The Last-Minute Life Saver**.

The backend is responsible for:

* Business Logic
* Authentication & Authorization
* Database Operations
* AI Agent Orchestration
* Event Management
* Background Jobs
* Notifications
* Analytics
* External Integrations

The backend **must never become a monolithic controller-based application**. Every responsibility should be isolated into dedicated services.

# Repository Layer

Repositories are responsible for database interaction only.


Repositories must not contain business logic.

# AI Orchestrator

The backend contains an **AI Orchestrator Service** responsible for coordinating AI agents.

Responsibilities:

* Receive AI requests
* Load user context
* Select required agents
* Execute agents in sequence
* Aggregate outputs
* Handle failures
* Cache responses

The orchestrator is the only component allowed to invoke AI agents.

# Event Bus

The backend should use an event-driven architecture.

Events include:

* TaskCreated
* TaskUpdated
* TaskCompleted
* GoalCreated
* GoalCompleted
* HabitCompleted
* ScheduleGenerated
* CalendarSynced
* ReminderSent
* UserLoggedIn

Events decouple services and enable background processing.

---

# Background Workers

Workers execute long-running or asynchronous tasks.

## Scheduler Worker

Runs daily or when schedules need regeneration.

Responsibilities:

* Generate daily schedules
* Resolve conflicts
* Rebalance workloads

---

## Reminder Worker

Runs every few minutes.

Responsibilities:

* Check upcoming events
* Send notifications
* Escalate reminders

---

## Learning Worker

Continuously updates productivity models.

Learns:

* Completion times
* Preferred work hours
* Focus duration
* Habit consistency

---

## Analytics Worker

Generates:

* Productivity scores
* Weekly summaries
* Monthly reports

---

## Calendar Worker

Synchronizes:

* Google Calendar ‚Üí MongoDB
* MongoDB ‚Üí Google Calendar

# Queue System

Use BullMQ with Redis.

Queues:

* ai-analysis
* schedule-generation
* notifications
* analytics
* calendar-sync
* learning-engine

Queues prevent blocking API responses and improve scalability.

---

# Error Handling Strategy

Every service must return standardized errors.

Example:

```json
{
  "success": false,
  "error": {
    "code": "TASK_NOT_FOUND",
    "message": "Requested task does not exist."
  }
}
```

Unexpected errors should be logged and handled by the global error middleware.

---

# Logging Strategy

Use structured logging with Pino.

Log:

* API requests
* Errors
* AI execution
* Background jobs
* Authentication events
* Database failures

Sensitive information must never be logged.

---

# Validation Strategy

All incoming requests must be validated using Zod.

Validation includes:

* Required fields
* Data types
* String lengths
* Date formats
* Enum values
* Business rules

No invalid request should reach the Service Layer.

---

# Security Guidelines

Implement:

* HTTPS
* JWT verification
* Helmet
* CORS
* Rate limiting
* Input sanitization
* Secure environment variables
* Password hashing (where applicable)
* Audit logs

---

# Caching Strategy

Redis should cache:

* User preferences
* Generated schedules
* AI responses
* Frequently accessed analytics
* Calendar snapshots

Cache invalidation must occur when relevant data changes.

---

# API Standards

All APIs should follow REST conventions.

Example:

```text
GET    /api/tasks
POST   /api/tasks
GET    /api/tasks/:id
PUT    /api/tasks/:id
DELETE /api/tasks/:id
```

Responses must use consistent formats.

Success:

```json
{
  "success": true,
  "data": {}
}
```

Failure:

```json
{
  "success": false,
  "error": {}
}
```

# Coding Standards

* Use TypeScript exclusively.
* Prefer async/await.
* Keep controllers lightweight.
* Business logic belongs in services.
* Database logic belongs in repositories.
* Avoid circular dependencies.
* Write unit tests for services.
* Use dependency injection where practical.

---

# Antigravity Development Instructions

When generating backend code:

1. Create the folder structure exactly as specified.
2. Implement middleware before feature services.
3. Build repositories before services.
4. Implement the AI Orchestrator before individual AI agents.
5. Use BullMQ for asynchronous processing.
6. Use Redis for caching and queues.
7. Keep controllers thin.
8. Never place business logic inside routes or controllers.
9. Follow REST conventions and standardized response formats.
10. Prepare the architecture for future microservice extraction.
