# Purpose

This document defines how every component in the system communicates with each other.

It specifies:

* Request Flow
* Response Flow
* Event Bus
* Queue System
* AI Communication
* Database Communication
* Background Workers
* Error Propagation
* Retry Mechanisms
* Service Contracts

The goal is to create a loosely coupled architecture where services communicate through events and APIs rather than direct dependencies.

---

# Communication Principles

All communication must follow these principles:

* Loose Coupling
* Single Responsibility
* Event Driven
* Asynchronous Where Possible
* Idempotent Operations
* Observable
* Fault Tolerant

# Communication Layers

## Layer 1 – Client Layer

Responsibilities:

* User interaction
* Display data
* API requests
* Real-time updates

Allowed communication:

Frontend → Backend API

Not allowed:

Frontend → Database

Frontend → AI

Frontend → Redis

---

## Layer 2 – API Layer

Responsibilities:

* Authentication
* Validation
* Authorization
* Request routing

No business logic should exist here.

---

## Layer 3 – Business Layer

Responsibilities:

* Business rules
* Service coordination
* AI orchestration
* Event publishing

---

## Layer 4 – AI Layer

Responsibilities:

* Reasoning
* Planning
* Scheduling
* Coaching
* Reminder generation

Agents never access the database directly.

---

## Layer 5 – Data Layer

Responsibilities:

* Persist data
* Query data
* Index management
* Transactions

# Event Bus Architecture

The backend uses an internal Event Bus.

Services publish events without knowing who consumes them.

Example:

```text
Task Created

↓

Event Bus

↓

Scheduler Worker

↓

Reminder Worker

↓

Analytics Worker

↓

Learning Worker
```

Advantages:

* Decoupled services
* Easier testing
* Better scalability
* Background processing

# Retry Strategy

If a service fails:

1. Retry once immediately.
2. Retry using exponential backoff.
3. Log failure.
4. Publish failure event.
5. Notify monitoring system.
6. Fall back if applicable.

---

# Timeout Policy

| Service      | Timeout |
| ------------ | ------- |
| Database     | 5 sec   |
| AI Request   | 30 sec  |
| Calendar API | 10 sec  |
| Notification | 5 sec   |

Operations exceeding timeout should be retried or gracefully degraded.

---

# Idempotency

Operations must be safe to retry.

Example:

* Sending reminders should not create duplicates.
* Completing a task twice should have no additional effect.
* Calendar synchronization should avoid duplicate events.

---

# Standard Service Contract

Every service should expose:

```typescript
interface ServiceResult<T> {
  success: boolean;
  data?: T;
  error?: {
    code: string;
    message: string;
  };
}
```

This ensures consistent responses throughout the system.

---

# Observability

Monitor:

* API latency
* AI execution time
* Queue length
* Worker failures
* Database queries
* Notification delivery
* Calendar sync success

Logs should include correlation IDs to trace requests across services.

---

# Error Propagation

Errors should flow upward in a controlled manner.

```text
Repository Error

↓

Service

↓

Controller

↓

API Response
```

Sensitive internal details must never be exposed to clients.

# Antigravity Development Instructions

When implementing service communication:

1. Use REST APIs for client requests.
2. Introduce an internal Event Bus for asynchronous processing.
3. Use BullMQ with Redis for background jobs.
4. Keep services independent.
5. Communicate through interfaces or events.
6. Validate every AI response before use.
7. Use typed contracts between services.
8. Make operations idempotent where possible.
9. Log every significant event with correlation IDs.
10. Avoid direct service-to-service dependencies unless necessary.
