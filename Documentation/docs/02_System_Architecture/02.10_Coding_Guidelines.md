# Purpose

This document defines the coding standards, architecture principles, naming conventions, testing requirements, and development workflow for the project.

Every contributor—human or AI—must follow these guidelines to ensure:

* Consistent code quality
* Maintainability
* Scalability
* Readability
* Testability
* Predictable architecture

---

# Core Engineering Principles

The project follows:

* Clean Architecture
* SOLID Principles
* DRY (Don't Repeat Yourself)
* KISS (Keep It Simple, Stupid)
* YAGNI (You Aren't Gonna Need It)
* Composition over Inheritance
* Feature-Based Organization

Every module should have a single, clearly defined responsibility.

# Code Style

## Formatting

* 2-space indentation
* UTF-8 encoding
* LF line endings
* Maximum line length: 100 characters
* Trailing commas where supported
* Semicolons required
* Single quotes for strings (unless framework requires otherwise)

Formatting should be enforced automatically using Prettier.

---

# Naming Conventions

## Files

Use **kebab-case**.

Examples:

```text
task.service.ts

planner.agent.ts

notification.worker.ts
```

---

## Classes

Use **PascalCase**.

```typescript
class TaskService {}

class PlannerAgent {}

class ReminderWorker {}
```

---

## Functions

Use **camelCase**.

```typescript
createTask()

generateSchedule()

sendReminder()

calculatePriority()
```

---

## Variables

Use descriptive camelCase names.

Good:

```typescript
estimatedDuration

remainingTime

productivityScore
```

Avoid:

```typescript
x

temp

data1
```

---

## Constants

Use **UPPER_SNAKE_CASE**.

```typescript
MAX_RETRY_ATTEMPTS

DEFAULT_BREAK_DURATION

API_TIMEOUT
```

---

## Interfaces

Prefix with `I` is **not** required.

Good:

```typescript
Task

UserPreferences

ScheduleResult
```

---

## Enums

Use PascalCase.

```typescript
enum TaskPriority {}

enum ReminderType {}
```

---

# Folder Organization Rules

* One feature per folder.
* One responsibility per file.
* Avoid files exceeding 300–400 lines.
* Split large modules into smaller units.

Example:

```text
tasks/

task.service.ts

task.repository.ts

task.controller.ts

task.validation.ts

task.events.ts

task.types.ts
```

---

# Architecture Rules

## Controllers

Controllers should:

* Validate requests
* Call services
* Return responses

Controllers must **not**:

* Query the database
* Call AI directly
* Contain business logic

---

## Services

Services contain:

* Business rules
* AI orchestration
* Event publishing

Services should never contain HTTP-specific logic.

---

## Repositories

Repositories:

* Query MongoDB
* Perform CRUD
* Map database models

Repositories should not contain business decisions.

---

## Workers

Workers:

* Consume queues
* Process background jobs
* Publish events

Workers should remain stateless.

---

## AI Agents

Each AI agent:

* Has one responsibility
* Receives structured input
* Produces structured output
* Does not access the database directly

# TypeScript Standards

Enable:

```json
{
  "strict": true,
  "noImplicitAny": true,
  "strictNullChecks": true,
  "noUnusedLocals": true,
  "noUncheckedIndexedAccess": true
}
```

Avoid:

* `any`
* `@ts-ignore`
* non-null assertions (`!`) unless justified

Prefer explicit types for public APIs.

---

# Error Handling

Use custom error classes.

Example:

```typescript
class TaskNotFoundError extends Error {}
```

Never throw raw strings.

Every error should include:

* Error code
* Message
* HTTP status (where applicable)

---

# API Response Standard

Success:

```json
{
  "success": true,
  "data": {}
}
```

Failure:

```json
{
  "success": false,
  "error": {
    "code": "TASK_NOT_FOUND",
    "message": "Task not found."
  }
}
```

Maintain this structure across all endpoints.

---

# Logging Standards

Log:

* Incoming requests
* AI execution
* Queue processing
* Errors
* Authentication events
* External API failures

Do **not** log:

* Passwords
* Tokens
* API keys
* Personal user content unless required for debugging and properly sanitized

Use structured JSON logging.

---

# Validation Standards

Validate all external input using Zod.

Validation includes:

* Required fields
* Data types
* Length constraints
* Date formats
* Enum values
* Business rules

Never trust client input.

---

# AI Coding Rules

AI-generated code must:

* Follow project architecture
* Use existing shared types
* Avoid duplicate logic
* Keep prompts external
* Validate AI responses
* Include error handling
* Be deterministic where possible

Never embed prompt text directly into service files.

---

# Testing Standards

Every feature should include:

* Unit tests
* Integration tests
* End-to-end tests (where appropriate)

Target coverage:

| Layer        | Coverage |
| ------------ | -------: |
| Services     |      90% |
| Utilities    |      95% |
| AI Schemas   |     100% |
| Repositories |      80% |
| Controllers  |      70% |

---

# Documentation Standards

Every public function should include:

* Purpose
* Parameters
* Return value
* Possible errors

Complex modules should include architecture comments at the top of the file.

---

# Git Workflow

Branch naming:

```text

feature/scheduler-agent

bugfix/calendar-sync

hotfix/auth-token

refactor/task-service
```

# Commit Convention

Use Conventional Commits.

Examples:

```text
feat: add planner agent

fix: resolve calendar sync conflict

refactor: simplify scheduler service

docs: add AI architecture specification

test: add task service unit tests

chore: update dependencies
```

---

# Pull Request Checklist

Every pull request should:

* Pass linting
* Pass tests
* Update documentation if needed
* Include meaningful commit messages
* Avoid unrelated changes
* Follow architecture guidelines

---

# Definition of Done (DoD)

A feature is considered complete only if:

* Requirements implemented
* Tests passing
* Linting passes
* Documentation updated
* Error handling implemented
* Logging added
* Validation added
* Code reviewed
* No critical TODOs remain

---

# Performance Guidelines

* Avoid unnecessary re-renders.
* Minimize database queries.
* Use indexes effectively.
* Cache frequently accessed data.
* Offload long-running work to workers.
* Batch AI requests when practical.

Measure before optimizing.

---

# Security Guidelines

* Sanitize inputs.
* Escape output where necessary.
* Verify JWTs.
* Enforce authorization checks.
* Use HTTPS.
* Store secrets in environment variables.
* Rotate credentials regularly.

Never expose internal implementation details in API responses.

---

# Code Review Checklist

Reviewers should verify:

* Correct architecture layer usage
* No duplicated logic
* Naming conventions followed
* Error handling present
* Validation implemented
* Tests included
* Documentation updated
* Performance considerations addressed
* Security best practices followed

---

# Antigravity Development Instructions

When generating code:

1. Follow Clean Architecture exactly.
2. Keep controllers thin.
3. Place business logic in services.
4. Use repositories for persistence.
5. Use TypeScript strict mode.
6. Never use `any`.
7. Validate all inputs with Zod.
8. Use structured logging.
9. Write tests alongside new features.
10. Follow Conventional Commits for generated commit messages.

---

# Engineering Philosophy

The project prioritizes:

1. Readability over cleverness.
2. Correctness over speed of implementation.
3. Simplicity over unnecessary abstraction.
4. Maintainability over short-term optimization.
5. Extensibility over rigid design.

Every architectural decision should support these principles.
