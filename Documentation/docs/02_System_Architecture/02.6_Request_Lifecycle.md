# Purpose

This document defines the complete request lifecycle for every major feature in the application.

For each user action, it specifies:

* Frontend interaction
* API endpoint
* Backend processing
* AI agent execution
* Database operations
* Background events
* Notifications
* Final response
* Error handling

This document serves as the execution blueprint for developers and AI coding agents.

---

# General Request Lifecycle

Every request follows the same high-level pipeline.

```text
User Action
    │
    ▼
Frontend
    │
    ▼
Authentication
    │
    ▼
Validation
    │
    ▼
Controller
    │
    ▼
Business Service
    │
    ▼
Repository
    │
    ▼
Database
    │
    ▼
Publish Events
    │
    ▼
AI Orchestrator (if required)
    │
    ▼
Background Workers
    │
    ▼
Response
```

# Error Handling

## Validation Error

Return immediately.

No database update.

---

## AI Failure

Fallback to:

* Cached result
* Rule-based logic

User is informed only if the failure affects functionality.

---

## Database Failure

Rollback transaction where applicable.

Log error.

Retry background jobs if necessary.

---

## External API Failure

Google Calendar unavailable:

* Retry
* Continue with cached events

Notification service unavailable:

* Retry via queue

---

# Lifecycle Principles

Every lifecycle must:

* Be idempotent where possible.
* Publish meaningful events.
* Keep controllers lightweight.
* Keep AI independent.
* Avoid blocking the user with long-running tasks.

# Antigravity Development Instructions

When implementing request lifecycles:

1. Follow the lifecycle sequence exactly.
2. Keep business logic in services.
3. Publish events after successful database writes.
4. Invoke AI only through the AI Orchestrator.
5. Offload long-running work to background queues.
6. Ensure every lifecycle has clear success and failure paths.
7. Log every major transition with correlation IDs.
8. Design all workflows to support retries and future extensions.
