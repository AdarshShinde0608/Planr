# 05.4 State Management Architecture

## Overview

This document defines the **global state management system** for The Last-Minute Life Saver.

It ensures:
- Predictable data flow
- AI state coordination
- Theme synchronization
- Task + schedule consistency
- Scalable frontend architecture

---

## Core Philosophy

State management follows 4 principles:

### 1. Single Source of Truth
All critical data exists in centralized stores.

### 2. Separation of Concerns
UI state, server state, and AI state are separated.

### 3. Reactive Updates
UI automatically reacts to state changes.

### 4. Minimal Global State
Only essential data is global.

---

## State Categories

### 1. UI State

Includes:

- Sidebar open/close
- Modal visibility
- Theme (light/dark)
- Active tab
- Navigation state

---

### 2. Server State

Includes:

- Tasks
- Projects
- Calendar events
- User profile
- Notifications

Handled via API + caching layer.

---

### 3. AI State

Includes:

- AI suggestions
- Generated schedules
- Planner outputs
- Chat context
- Agent memory snapshot

---

### 4. Local Component State

Includes:

- Input fields
- Form states
- Temporary UI interactions
- Animations

---

## Recommended Architecture

### State Stack

```text id="state02"
UI State        → Zustand / Redux
Server State    → React Query / TanStack Query
AI State        → Custom AI Store Layer
Local State     → React useState
``` 

## Global State Stores
### 1. UI Store

Manages:

Theme mode
Sidebar state
Modal system
Layout mode

Example structure:
```typescript
interface UIState {
  theme: "dark" | "light";
  sidebarOpen: boolean;
  activeRoute: string;
  isCommandPaletteOpen: boolean;
}
```

### 2. Task Store

Manages:

Task creation
Task updates
Priority system
Deadline tracking
```typescript
interface Task {
  id: string;
  title: string;
  description?: string;
  priority: "low" | "medium" | "high";
  deadline: Date;
  status: "todo" | "in-progress" | "done";
}
```

### 3. AI Store

Manages:

AI-generated schedules
Suggestions
Planner output
Chat context
```typescript
interface AIState {
  suggestions: string[];
  generatedSchedule: any;
  contextWindow: any[];
  lastUpdated: Date;
}
```

## Server State Strategy
Use React Query (TanStack Query)

Benefits:
Caching
Background sync
Auto refetch
Reduced API calls
Example Query
```typescript
useQuery({
  queryKey: ["tasks"],
  queryFn: fetchTasks,
});
```

## Theme State Integration

Theme state is globally managed:

Stored in UI Store
Synced with localStorage
Connected to DOM attribute
UI Store → localStorage → <html data-theme>

## AI State Handling
AI state is treated separately because:

It is asynchronous
It evolves dynamically
It depends on user context
AI Flow
User Input
   ↓
AI Orchestrator
   ↓
Agent Processing
   ↓
AI Store Update
   ↓
UI Re-render


## State Synchronization Rules
Rule 1

Never duplicate server state locally.

Rule 2

AI state must not override server truth.

Rule 3

UI state must not persist business data.

## Performance Optimization
Use memoization for derived state
Avoid unnecessary global updates
Split stores by domain
Use shallow selectors

## Recommended Libraries
Purpose	Library
UI State	Zustand
Server State	React Query
Complex State	Redux Toolkit (optional)
Forms	React Hook Form

## Event Flow Architecture
User Action
   ↓
UI Store Update
   ↓
API Call (if needed)
   ↓
Server Response
   ↓
React Query Update
   ↓
UI Re-render

## Error State Handling
All stores must support:

Loading states
Error states
Retry mechanisms
Fallback UI

## Persistence Strategy
Only persist:

Theme
User preferences
UI layout settings

Do NOT persist:

Temporary AI outputs
UI transient states
Cache-only data

## Security Considerations
Never store sensitive data in UI store
Avoid exposing AI context publicly
Validate all server updates
