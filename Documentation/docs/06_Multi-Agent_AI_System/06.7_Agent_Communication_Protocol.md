# Purpose

This document defines the communication standards used by all AI agents.

The goals are:

* Standardized communication
* Independent deployment
* Reliable workflows
* Easy scalability
* Strong observability
* Fault tolerance
* Version compatibility

Every AI agent must follow this protocol.

---

# Communication Philosophy

Agents should **never communicate directly**.

Instead:

```text
Planner Agent

↓

AI Orchestrator

↓

Scheduler Agent
```

Advantages:

* Loose coupling
* Better monitoring
* Easier testing
* Retry support
* Security
* Future scalability

---


# Standard Request Schema

Every request follows the same structure.

```json
{
  "requestId": "uuid",
  "correlationId": "uuid",
  "workflowId": "daily_schedule",
  "timestamp": "2026-07-02T12:00:00Z",
  "agent": "planner",
  "context": {},
  "payload": {}
}
```

---

# Standard Response Schema

```json
{
  "requestId": "uuid",
  "correlationId": "uuid",
  "agent": "planner",
  "status": "SUCCESS",
  "executionTime": 523,
  "confidence": 0.94,
  "data": {},
  "errors": []
}
```

Every agent must return this structure.

---

# Correlation ID

Every workflow receives a unique Correlation ID.

Example:

```text
User

↓

Create Schedule

↓

Correlation ID

ABC-12345
```

All agents processing that workflow use the same ID.

Benefits:

* Distributed tracing
* Debugging
* Monitoring
* Performance analysis

---

# Workflow ID

Each workflow has its own identifier.

Examples:

```text
daily_schedule

task_analysis

goal_planning

habit_analysis

reschedule_day

deadline_alert
```

# Context Propagation

Every agent receives the same standardized context.

```json
{
  "user": {},
  "tasks": [],
  "schedule": {},
  "calendar": {},
  "preferences": {},
  "analytics": {},
  "history": {}
}
```

No agent should request missing context directly.

The AI Orchestrator is responsible for context loading.

# Agent Communication Rules

Allowed:

```text
Planner

↓

AI Orchestrator

↓

Scheduler
```

Not allowed:

```text
Planner

↓

Scheduler
```

Agents remain unaware of each other's implementation.

---

# Synchronous Communication

Use when:

* Immediate response required
* User waiting

Examples:

* Add Task
* Generate Schedule
* AI Chat

---

# Asynchronous Communication

Use when:

* Background work
* Analytics
* Daily summaries
* Learning
* Weekly reports

Example:

```text
Task Completed

↓

Publish Event

↓

Coach Updates Analytics

↓

Reminder Learns Behavior
```

# Response Validation

Before returning:

* Validate schema
* Validate required fields
* Validate confidence
* Validate timestamps

Invalid responses should never reach users.

---

# Error Response

Standard format:

```json
  "message": "Planner Agent timed out.",
  "retryable": true
```

```typescript

    request: AgentRequest

}
```

---

# Best Practices

* Never allow direct agent-to-agent communication.
* Use immutable message contracts.
* Validate all requests and responses.
* Keep messages lightweight.
* Include correlation IDs for tracing.
* Prefer asynchronous events for background work.
* Centralize routing in the AI Orchestrator.
* Log every communication event.

---

# Antigravity Development Instructions

When implementing the Agent Communication Protocol:

1. Route all agent communication through the AI Orchestrator.
2. Implement standardized request and response schemas shared by every AI agent.
3. Generate unique Request IDs and Correlation IDs for every workflow.
4. Support synchronous communication for user-facing operations and asynchronous communication for background processing.
5. Validate all messages against predefined schemas before processing.
6. Implement retry, timeout, and failure recovery policies.
7. Introduce an internal Event Bus abstraction that can later be replaced with Kafka or RabbitMQ.
8. Record detailed logs and distributed traces for every workflow.
9. Keep communication protocol versioned for future compatibility.
10. Ensure agents remain fully decoupled and independently deployable.
