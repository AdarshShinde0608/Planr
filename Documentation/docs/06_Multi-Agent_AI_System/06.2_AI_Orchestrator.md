# Purpose

The AI Orchestrator is the **central coordinator** of the Multi-Agent AI system.

It is responsible for:

* Receiving AI requests
* Understanding user intent
* Selecting appropriate agents
* Loading context
* Coordinating execution
* Aggregating results
* Handling failures
* Returning a unified AI response

The orchestrator **never performs domain reasoning** itself. All reasoning is delegated to specialized agents.

---

# Responsibilities

The orchestrator is responsible for:

* Request routing
* Agent selection
* Context management
* Workflow execution
* Parallel task coordination
* Dependency resolution
* Result aggregation
* Error handling
* Logging
* Performance monitoring

# Intent Detection

The orchestrator identifies the primary objective before invoking agents.

Supported intents include:

| Intent                     | Primary Agent   |
| -------------------------- | --------------- |
| Create Task                | Planner         |
| Analyze Task               | Planner         |
| Generate Schedule          | Scheduler       |
| Reschedule Day             | Scheduler       |
| Productivity Advice        | Coach           |
| Habit Guidance             | Coach           |
| Reminder Request           | Reminder        |
| Mixed Productivity Request | Multiple Agents |

Multiple intents may be detected from a single request.

# Context Loading

Before invoking any agent, the orchestrator loads the relevant user context.

Possible context:

* User profile
* Daily routine
* Calendar events
* Active schedule
* Tasks
* Goals
* Habits
* Previous AI interactions
* Productivity statistics
* Notification preferences

Only the required context should be loaded for efficiency.

# Execution Modes

## Sequential

Use when later agents depend on earlier outputs.

---

## Parallel

Run independent agents simultaneously.

Advantages:

* Lower latency
* Better scalability

---

## Conditional

Invoke an agent only when conditions are met.

Example:

```text
Task Deadline < 24 Hours

↓

Reminder Agent
```

---

## Event-Driven

Triggered by system events.

Examples:

* Calendar updated
* Task completed
* Goal achieved
* Habit skipped

# Conflict Resolution

Conflicts may occur.

Example:

Planner:

```text
Estimated Time = 6 Hours
```

Scheduler:

```text
Only 3 Hours Available
```

Resolution strategy:

1. Detect conflict.
2. Request Scheduler optimization.
3. If unresolved, notify the user with alternatives.

---

# Failure Recovery

Agent failures should not crash the entire system.

Example:

Planner ✔

Scheduler ❌

Coach ✔

Reminder ✔

Return partial response:

> "Task analysis is available. Scheduling is temporarily unavailable."

---

# Timeout Strategy

Each agent has an execution timeout.

Example:

| Agent     | Timeout |
| --------- | ------- |
| Planner   | 10 sec  |
| Scheduler | 12 sec  |
| Coach     | 8 sec   |
| Reminder  | 5 sec   |

Timed-out agents return fallback responses.

---

# Retry Policy

Retry transient failures:

1. Immediate retry
2. Exponential backoff
3. Mark agent unavailable if retries fail

Permanent failures should be logged.

# Best Practices

* Keep orchestration logic independent of business logic.
* Never allow agents to call each other directly.
* Share structured context only.
* Prefer parallel execution where dependencies allow.
* Gracefully degrade when agents fail.
* Log every workflow execution.
* Keep workflows configurable rather than hard-coded.

---

# Antigravity Development Instructions

When implementing the AI Orchestrator:

1. Create a dedicated **AI Orchestrator Service** that acts as the single entry point for all AI workflows.
2. Implement intent detection before selecting agents.
3. Load only the context required for the selected workflow.
4. Support sequential, parallel, conditional, and event-driven execution modes.
5. Inject structured context into agents rather than allowing direct database access.
6. Aggregate structured agent responses into a unified result.
7. Implement timeout, retry, and graceful failure handling.
8. Record execution metrics and logs for monitoring.
9. Design workflows to be configurable through metadata rather than hard-coded logic.
10. Ensure new agents can be added without modifying existing orchestrator code.
