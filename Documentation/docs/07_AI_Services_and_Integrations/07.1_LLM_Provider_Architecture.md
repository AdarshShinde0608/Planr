# Purpose

This document defines the architecture responsible for interacting with Large Language Models (LLMs).

Instead of allowing AI agents to communicate directly with a specific provider (OpenAI, Gemini, Anthropic, etc.), every request passes through a centralized **LLM Provider Layer**.

This abstraction provides:

* Provider independence
* Automatic failover
* Cost optimization
* Model selection
* Rate limiting
* Logging
* Monitoring
* Prompt version control

---

# Objectives

The LLM Provider Layer should:

* Support multiple LLM providers
* Select the optimal model for each task
* Reduce operational costs
* Improve reliability
* Support future providers
* Centralize API management
* Simplify AI agent development

---

The LLM Provider Layer manages:

* Model routing
* Authentication
* Prompt execution
* Response validation
* Retry logic
* Failover
* Token tracking
* Cost calculation
* Request logging

# Supported Providers

Initial providers:

| Provider      | Purpose                 |
| ------------- | ----------------------- |
| OpenAI        | Primary reasoning model |
| Google Gemini | Large context tasks     |
| Anthropic     | Long-form reasoning     |
| Groq          | Low-latency inference   |
| Ollama        | Local/offline inference |

Future providers can be added without modifying AI agents.

| Model         | Use Case                 |
| ------------- | ------------------------ |
| GPT-5.5       | Planner                  |
| GPT-5.5 Mini  | Reminder                 |
| Gemini Pro    | Large-context scheduling |
| Claude        | Coaching and summaries   |
| Llama (Local) | Offline inference        |

The registry should be configuration-driven.

# Model Selection Strategy

Choose models based on:

* Task complexity
* Context length
* Required latency
* Cost
* Availability
* User subscription tier

Example:

| Task                | Recommended Model     |
| ------------------- | --------------------- |
| Task Classification | Small model           |
| Schedule Generation | Large reasoning model |

# Provider Failover

If the primary provider fails:

```text
OpenAI

↓

Timeout

↓

Retry

↓

Gemini

↓

Success
```

Fallback order should be configurable.

---

# Retry Policy

Retry sequence:

1. Immediate retry
2. Exponential backoff
3. Switch provider
4. Return graceful error

Never retry indefinitely.

# Response Validation

Every response must pass:

* JSON parsing
* Schema validation
* Business rule validation
* Confidence validation

Invalid responses are rejected and retried.

* Input tokens

* Total tokens
* Average tokens/request

---

# Cost Tracking

Record:

* Provider
* Model
* Token usage
* Estimated cost
* Request timestamp

Daily and monthly reports should be generated.

---

# Rate Limiting

Protect providers from excessive requests.

Example:

| Agent     | Requests / Minute |
| --------- | ----------------: |
| Planner   |                20 |
| Scheduler |                10 |
| Coach     |                 5 |
| Reminder  |                30 |

Queue excess requests when necessary.

---

# Prompt Caching

Cache deterministic responses.

Suitable for:

* Task classification
* Similar planning requests
* Repeated schedule generation inputs

Avoid caching personalized coaching responses.

---

# Streaming Support

Support streaming responses for:

* AI Chat Assistant
* Voice Assistant
* Long reflections

Planner and Scheduler generally return complete JSON responses.

---

# Observability

Monitor:

# Security

The LLM Provider Layer must:

* Secure API keys using a secrets manager.
* Never expose provider credentials.
* Encrypt request logs where necessary.
* Mask sensitive user information before sending prompts.
* Support per-provider access control.

# Best Practices

* Never hardcode provider-specific logic inside AI agents.
* Separate provider selection from prompt construction.
* Validate every LLM response before use.
* Track token usage and costs continuously.
* Cache deterministic responses where appropriate.
* Support graceful degradation during outages.
* Keep provider integrations modular and replaceable.

---

# Antigravity Development Instructions

When implementing the LLM Provider Layer:

1. Create a dedicated **LLM Service** accessed exclusively through the AI Orchestrator.
2. Implement adapters for each supported provider using a common interface.
3. Build a configurable Model Registry to map AI tasks to the most appropriate model.
4. Add automatic retry and provider failover mechanisms.
5. Validate all LLM outputs using the schemas defined in Phase 6.
6. Record token usage, latency, costs, and provider metrics for every request.
7. Support both synchronous and streaming responses.
8. Integrate prompt caching for deterministic workflows to reduce latency and cost.
9. Store API credentials securely using environment variables or a secrets manager.
10. Design the provider layer so additional LLM vendors can be added without modifying existing AI agents.
