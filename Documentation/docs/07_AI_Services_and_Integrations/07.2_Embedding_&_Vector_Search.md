# Purpose

The Embedding & Vector Search System enables the AI to understand semantic meaning instead of relying on exact keyword matches.

Rather than searching only for identical text, the system retrieves memories, tasks, goals, and user behavior that are conceptually similar.

Example:

Stored Task:

> Finish Database Assignment

User asks:

> I need to complete my DBMS homework.

Even though the wording is different, vector similarity identifies them as related.

---

# Objectives

The system should:

* Store semantic embeddings
* Retrieve similar memories
* Improve AI personalization
* Support natural language search
* Reduce prompt size
* Enhance recommendation quality
* Scale efficiently

# Why Embeddings?

Traditional SQL:

```text
Task = "Database Assignment"

Search:

"DBMS Homework"

↓

No Match
```

Vector Search:

```text
Database Assignment

≈

DBMS Homework

↓

High Similarity
```

# Content Not to Embed

Do NOT generate embeddings for:

* Passwords
* Authentication tokens
* Email verification codes
* API keys
* Sensitive credentials
* Binary attachments

---

# Embedding Model

Requirements:

* High semantic accuracy
* Low latency
* Small vector size
* Versioned outputs

Configuration example:

```yaml
embedding:
  provider: OpenAI
  model: text-embedding-3-small
  dimensions: 1536
```

The model should be configurable to allow future upgrades.

---

# Vector Storage

Recommended database:

**PostgreSQL + pgvector**

Benefits:

* Native PostgreSQL integration
* ACID compliance
* SQL + vector search
* Easy backups
* Simple deployment

---

# Database Schema

```sql
CREATE TABLE embeddings (
    id UUID PRIMARY KEY,
    user_id UUID,
    source_type TEXT,
    source_id UUID,
    embedding VECTOR(1536),
    metadata JSONB,
    created_at TIMESTAMP
);
```

---

# Metadata

Each vector stores metadata.

Example:

```json
{
  "sourceType": "Task",
  "priority": "High",
  "category": "College",
  "createdBy": "PlannerAgent"
}
```

Metadata enables hybrid filtering.

# Similarity Search

Supported metrics:

* Cosine Similarity (recommended)
* Euclidean Distance
* Inner Product

Default:

**Cosine Similarity**

# Top-K Retrieval

Retrieve only the most relevant items.

Default:

```text
Top K = 5
```

Configurable:

* 3
* 5
* 10
* 20

Avoid excessive context.

---

# Hybrid Search

Combine SQL filtering with vector similarity.

Example:

SQL:

```text
Category = College
```

Then:

Vector Search:

```text
"Database Homework"
```

This improves accuracy and reduces irrelevant matches.

---

# Re-ranking

Initial vector results should be re-ranked using:

* Similarity score
* Recency
* User interaction frequency
* Confidence
* Importance

Final ranking:

```text
Vector Score

+

Recency

+

Priority

+

Confidence

↓

Final Rank
```

---

# Context Injection

Only the highest-ranked results are passed to the AI.

Example:

```text
Retrieve 20

↓

Rank

↓

Select Best 5

↓

Inject into Prompt
```

# Best Practices

* Use cosine similarity as the default metric.
* Keep embeddings synchronized with source data.
* Limit prompt context to the highest-ranked results.
* Combine SQL filters with vector search.
* Version every embedding model.
* Cache frequently reused embeddings.
* Monitor retrieval quality continuously.

---

# Antigravity Development Instructions

When implementing the Embedding & Vector Search System:

1. Use **PostgreSQL with pgvector** as the primary vector database.
2. Build an Embedding Service responsible for generating, updating, deleting, and searching embeddings.
3. Generate embeddings for tasks, goals, habits, user preferences, and AI-generated insights.
4. Store structured metadata alongside each vector to enable hybrid search.
5. Implement cosine similarity as the default search metric.
6. Retrieve the Top-K most relevant vectors and apply a re-ranking pipeline before injecting context into AI prompts.
7. Support embedding versioning to allow future model upgrades without data loss.
8. Cache frequently accessed embeddings to reduce latency and API costs.
9. Secure all embeddings through encryption and user-level isolation.
10. Continuously monitor retrieval performance, relevance, and embedding lifecycle.
