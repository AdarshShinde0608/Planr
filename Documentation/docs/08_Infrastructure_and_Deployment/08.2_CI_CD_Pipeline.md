# Purpose

The CI/CD Pipeline automates the entire software delivery lifecycle, from code commits to production deployment.

Every code change should be:

* Validated
* Tested
* Built
* Scanned
* Containerized
* Deployed
* Monitored

This ensures high code quality, faster releases, and reliable deployments.

---

# Objectives

The CI/CD pipeline should:

* Automate builds
* Run tests
* Enforce code quality
* Detect security vulnerabilities
* Build Docker images
* Deploy automatically
* Support rollback
* Notify developers

# Best Practices

* Automate every repeatable task.
* Prevent direct production deployments.
* Keep builds reproducible.
* Use immutable Docker images.
* Enforce branch protection.
* Run automated security scans.
* Version every release.
* Monitor deployments continuously.

---

# Antigravity Development Instructions

When implementing the CI/CD pipeline:

1. Use **GitHub Actions** as the primary CI/CD platform.
2. Implement automated workflows for linting, formatting, unit tests, integration tests, and production builds.
3. Build and publish Docker images for every deployable service.
4. Scan dependencies and Docker images for vulnerabilities before deployment.
5. Deploy automatically to the staging environment after successful builds.
6. Require manual approval before deploying to production.
7. Execute database migrations as part of the deployment pipeline with rollback support.
8. Store all secrets using GitHub Secrets or a dedicated secret management solution.
9. Integrate deployment notifications with Slack, Discord, or email.
10. Validate health endpoints after deployment and automatically rollback if critical services fail.
