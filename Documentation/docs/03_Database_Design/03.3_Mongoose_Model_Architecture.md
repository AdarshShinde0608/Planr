# Purpose

This document defines the implementation architecture for all MongoDB models using Mongoose.

The objectives are:

* Maintain consistency across models
* Reduce duplicate code
* Standardize validation
* Automate audit fields
* Support soft deletes
* Improve query performance
* Simplify repository implementation

Every model in the application must follow this architecture.

---

# Model Architecture Overview

```text
REST API

↓

Controller

↓

Service

↓

Repository

↓

Mongoose Model

↓

MongoDB
```

The Repository Layer is the **only layer** allowed to directly access Mongoose models.

---

# Model Responsibilities

A model should only handle:

* Schema definition
* Validation
* Indexes
* Middleware
* Virtual fields
* Query helpers

A model should **not** contain:

* Business logic
* AI logic
* Scheduling logic
* Notification logic

# Base Schema

Every business model extends a common base schema.

```typescript
{
  createdAt: Date,
  updatedAt: Date,
  createdBy: ObjectId,
  updatedBy: ObjectId,
  isDeleted: Boolean,
  deletedAt: Date | null,
  version: Number
}
```

This ensures all collections share the same audit fields.

---

# Common Schema Options

Every schema should use:

```typescript
{
  timestamps: true,
  versionKey: "version",
  minimize: false
}
```

Benefits:

* Automatic timestamps
* Optimistic concurrency
* Consistent document structure

---

# Base Plugins

The application uses reusable Mongoose plugins.

## Soft Delete Plugin

Responsibilities:

* Add `isDeleted`
* Add `deletedAt`
* Override default queries
* Prevent deleted documents from appearing

---

## Audit Plugin

Automatically populates:

* createdAt
* updatedAt
* createdBy
* updatedBy

---

## Pagination Plugin

Provides reusable pagination helpers.

Example:

```typescript
TaskModel.paginate(...)
```

---

# Schema Validation

Validation should occur at two levels:

| Layer    | Responsibility         |
| -------- | ---------------------- |
| Zod      | Request validation     |
| Mongoose | Persistence validation |

Never rely on only one validation layer.

---

# Index Strategy

Indexes should be defined within the schema.

Example:

```typescript
taskSchema.index({
  userId: 1,
  deadline: 1
});
```

Avoid creating indexes dynamically during runtime.

---

# Virtual Fields

Use virtuals for computed values.

Example:

```typescript
taskSchema.virtual("isOverdue");
```

Possible virtuals:

* isOverdue
* completionPercentage
* remainingTime
* totalDuration

Virtuals should never be persisted.

---

# Middleware

## Pre-save Middleware

Used for:

* Normalizing strings
* Updating timestamps
* Cleaning data
* Generating slugs (future)

Example:

```typescript
taskSchema.pre("save", ...)
```

---

## Post-save Middleware

Should only perform lightweight actions.

Examples:

* Publish domain event
* Log activity

Heavy processing should be delegated to background workers.

---

# Query Helpers

Reusable query helpers improve readability.

Examples:

```typescript
Task.find().active()

Task.find().completed()

Task.find().pending()

Task.find().overdue()
```

Query helpers should encapsulate common filters.

---

# Static Methods

Static methods operate on collections.

Examples:

```typescript
Task.findByUser()

Task.findUpcoming()

Task.findOverdue()
```

Use statics for reusable collection queries.

---

# Instance Methods

Instance methods operate on a single document.

Examples:

```typescript
task.markCompleted()

task.archive()

task.calculateProgress()
```

Keep instance methods lightweight.

---

# Soft Delete Implementation

Deleting a task should:

```text
DELETE Request

↓

Repository

↓

Update:

isDeleted = true

deletedAt = now()

↓

Return Success
```

The document remains in the database for recovery and analytics.

---

# Optimistic Concurrency

Enable document versioning.

Every update increments:

```text
version
```

If two users edit the same document simultaneously, version conflicts can be detected and resolved.

---

# Lean Queries

Use `.lean()` for read-only operations.

Example:

```typescript
Task.find().lean();
```

Benefits:

* Faster execution
* Lower memory usage
* Better API performance

Avoid `.lean()` when middleware or document methods are required.

---

# Transactions

Repositories should use MongoDB transactions when updating multiple collections.

Examples:

* Task creation + activity log
* Goal completion + analytics update
* Schedule regeneration + notification queue

Keep transactions short to reduce lock contention.

---

# Model Registration

Register all models in a single location.

```text
models/

index.ts
```

Purpose:

* Centralized imports
* Prevent duplicate registrations
* Simplify testing

---

# Repository Integration

Repositories should expose methods such as:

```typescript
create()

findById()

findByUser()

update()

softDelete()

restore()

paginate()
```

Controllers and services should never call Mongoose models directly.

---

# Performance Guidelines

* Use projections to fetch only required fields.
* Avoid unnecessary population.
* Use indexes for frequently queried fields.
* Prefer references over deep embedding.
* Keep document size below MongoDB limits.
* Use bulk operations where appropriate.

---

# Error Handling

Convert Mongoose errors into domain-specific errors.

Example:

```text
Duplicate Key Error

↓

EmailAlreadyExistsError
```

The service layer should never expose raw database errors.

---

# Testing Guidelines

Each model should include tests for:

* Schema validation
* Required fields
* Default values
* Index creation
* Query helpers
* Soft delete behavior
* Middleware execution

Use an in-memory MongoDB instance for isolated model tests.

---

# Best Practices

* One model per file.
* Keep schema definitions focused.
* Use plugins for reusable behavior.
* Avoid business logic inside models.
* Validate all persisted data.
* Keep indexes synchronized with query patterns.
* Document custom statics and methods.

---

# Antigravity Development Instructions

When implementing Mongoose models:

1. Create a reusable `BaseSchema`.
2. Implement `SoftDeletePlugin`, `AuditPlugin`, and `PaginationPlugin`.
3. Apply common schema options to every model.
4. Define indexes within schema files.
5. Use virtuals only for computed values.
6. Keep middleware lightweight and delegate heavy tasks to workers.
7. Expose reusable query helpers, statics, and instance methods.
8. Use repositories as the only interface to models.
9. Write unit tests for every model.
10. Ensure all models conform to the shared architecture.
