# Purpose

This document defines the complete database architecture for **The Last-Minute Life Saver**.

The database is designed to support:

* User Management
* Task Management
* AI Multi-Agent System
* Dynamic Scheduling
* Calendar Integration
* Analytics
* Habit Tracking
* Goal Planning
* Notification System
* AI Learning Engine
* Future Team Collaboration

The database prioritizes:

* Scalability
* Performance
* Data Integrity
* Flexibility
* AI Context Retrieval
* Fast Scheduling Queries

# Why MongoDB?

MongoDB is selected because:

* Flexible schema
* Nested document support
* Excellent JSON compatibility
* Fast iteration during development
* Efficient handling of AI-generated metadata
* Easy horizontal scaling
* Suitable for dynamic scheduling data

# Database Architecture

```text
Frontend

↓

Express API

↓

Repository Layer

↓

MongoDB Atlas

↓

Redis Cache

↓

BullMQ Queues
```

Only the Repository Layer communicates directly with MongoDB.

# Design Principles

The database follows:

* Document-Oriented Design
* Normalization where appropriate
* Controlled denormalization for performance
* Reference-based relationships
* Immutable audit records
* Soft deletes
* Event-driven updates

---

# Soft Delete Strategy

Documents are never permanently removed.

Instead:

```json
{
  "isDeleted": true,
  "deletedAt": "2026-07-05T09:30:00Z"
}
```

Benefits:

* Recovery
* Analytics
* Audit history

---

# Relationships

MongoDB references are preferred over excessive embedding.

Example:

```text
Task

↓

userId

↓

Users Collection
```

Avoid embedding large datasets.

---

# Embedded Documents

Use embedding only for:

* Subtasks
* Reminder history
* Small AI metadata
* Daily schedule blocks

Example:

```json
{
  "subtasks": [
    {
      "title": "Research",
      "completed": false
    }
  ]
}
```

---

# Referenced Documents

Use references for:

* Tasks
* Goals
* Habits
* Analytics
* Calendar Events
* Notifications

This prevents oversized documents.

---

# AI Data Storage

AI-generated information should be stored separately from user-generated content.

Example:

```text
Task

↓

AI Metadata

↓

Planner Output
```

AI metadata includes:

* Estimated duration
* Predicted difficulty
* Deadline risk
* Suggested category
* Confidence score

This separation allows AI models to evolve without changing user data.

# Caching Strategy

Redis should cache:

* Dashboard data
* Daily schedules
* User preferences
* AI profile
* Productivity score
* Calendar snapshots

Cache expiration should be configurable.

---

# Indexing Strategy

Indexes are required for frequently queried fields.

Examples:

```text
users.email

tasks.userId

tasks.deadline

tasks.status

schedules.userId

calendarEvents.startTime

notifications.userId

analytics.userId
```

Composite indexes:

```text
(userId, deadline)

(userId, status)

(userId, createdAt)
```

---

# Transactions

Use MongoDB transactions for operations affecting multiple collections.

Examples:

* Task creation + event log
* Goal completion + analytics update
* Schedule regeneration + notification queue

Transactions should be limited to necessary scenarios.

---

# Audit Logging

Critical operations should generate audit records.

Tracked actions:

* Login
* Task creation
* Task completion
* Goal updates
* Preference changes
* Calendar sync
* AI schedule generation

Audit logs are append-only.

---

# Event Log Storage

Every published domain event should be recorded.

Example:

```json
{
  "eventType": "TaskCreated",
  "timestamp": "...",
  "userId": "...",
  "correlationId": "...",
  "status": "Processed"
}
```

Useful for debugging and replaying workflows.

---

# Performance Guidelines

* Keep documents small.
* Avoid deep nesting.
* Use projections for large queries.
* Index all frequently filtered fields.
* Archive historical analytics periodically.
* Avoid unbounded arrays in documents.

---

# Backup Strategy

MongoDB Atlas backups:

* Daily snapshots
* Point-in-time recovery
* Retention based on deployment environment

Critical collections should support disaster recovery.

---

# Security

* Encrypt data in transit (TLS).
* Encrypt data at rest.
* Restrict database access using least privilege.
* Store credentials in environment variables.
* Never expose internal IDs through public APIs unless required.

---

# Scalability

The database should support:

* Millions of tasks
* Millions of reminders
* Large analytics datasets
* Horizontal scaling through sharding (future)
* Read replicas for reporting workloads

No schema redesign should be required for future growth.

---

# Best Practices

* Use references for large relationships.
* Embed only small, bounded datasets.
* Prefer indexed queries.
* Keep AI metadata separate from user input.
* Enable soft deletes.
* Version documents where necessary.
* Maintain immutable audit records.

---

# Antigravity Development Instructions

When implementing the database:

1. Create collections exactly as defined.
2. Implement shared audit fields across all models.
3. Use references instead of large embedded documents.
4. Create indexes during schema initialization.
5. Separate AI-generated metadata from user-entered data.
6. Enable soft deletes for all business entities.
7. Use transactions for multi-collection operations.
8. Cache frequently accessed data in Redis.
9. Keep schemas extensible for future collaboration features.
10. Document every schema with field descriptions and validation rules.
