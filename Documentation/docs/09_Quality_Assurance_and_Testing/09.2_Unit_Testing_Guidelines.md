# Purpose

This document defines the standards, conventions, and best practices for writing unit tests throughout **The Last-Minute Life Saver**.

Every unit test should be:

* Independent
* Fast
* Deterministic
* Readable
* Easy to maintain

The objective is to detect defects as early as possible while keeping the test suite reliable and efficient.

---

# Objectives

Unit tests should:

* Verify individual units of code
* Run without external services
* Execute quickly
* Be deterministic
* Support refactoring
* Improve developer confidence

---

# Scope

Unit testing applies to:

* Utility functions
* Business services
* Controllers (mocked dependencies)
* React components
* Custom hooks
* Validation logic
* AI algorithms
* Scheduling algorithms
* Priority engine
* Helper libraries

Unit tests **must not** call:

* PostgreSQL
* Redis
* Firebase
* OpenAI
* Google Calendar
* External APIs

These dependencies should always be mocked.

---

# Unit Testing Principles

Follow these principles:

* Test one behavior per test.
* Keep tests isolated.
* Avoid shared mutable state.
* Prefer readability over cleverness.
* Test behavior, not implementation details.
* Make failures easy to understand.

---

# Arrange–Act–Assert Pattern

Every test should follow:

```text
Arrange
↓

Act
↓

Assert
```

Example

```typescript
describe("PriorityService", () => {

  it("should assign HIGH priority for overdue tasks", () => {

    // Arrange

    const task = createTask({
      deadline: yesterday
    });

    // Act

    const priority = service.calculate(task);

    // Assert

    expect(priority).toBe("HIGH");

  });

});
```

---

# Test Naming Convention

Use descriptive names.

Preferred

```text
should calculate priority for overdue task
```

Avoid

```text
test1

priority test

works correctly
```

One test file per implementation file.

# Mocking Strategy

Always mock:

* Database
* Redis
* Queue
* AI Providers
* Calendar API
* Notification Service
* Authentication Provider

Never depend on live services.

---

# Dependency Injection Testing

Use dependency injection to replace services.

Example

```typescript
const service = new PlannerService(

mockRepository,

mockAI,

mockScheduler

);
```

---

# Test Fixtures

Create reusable fixtures.

Example

```typescript
createUser()

createTask()

createGoal()

createSchedule()
```

Fixtures improve consistency and reduce duplication.

---

# Test Data

Use realistic data.

Good

```text
Complete DBMS Assignment

Deadline

Tomorrow
```

Avoid meaningless values such as:

```text
ABC

123

test
```

---

# Assertions

Prefer precise assertions.

Good

```typescript
expect(priority).toBe("HIGH");
```

Avoid vague assertions.

```typescript
expect(priority).toBeTruthy();
```

---

# Error Testing

Verify expected failures.

Example

```typescript
expect(() => {

service.calculate(undefined);

}).toThrow();
```

---

# Async Testing

Use async/await.

Example

```typescript
await service.generateSchedule();
```

Never rely on arbitrary delays.

---

# Time-Based Testing

Freeze time using fake timers.

Example

```typescript
jest.useFakeTimers();
```

This makes scheduling logic deterministic.

---

# AI Service Testing

Never call an LLM directly.

Mock responses.

Example

```typescript
mockAI.analyzeTask.mockResolvedValue({

priority: "HIGH",

duration: 120

});
```

Validate:

* Prompt creation
* Output parsing
* Error handling

Every unit should test:

* Empty input
* Null values
* Undefined values
* Invalid values
* Maximum limits
* Minimum limits

# Performance

Unit tests should complete quickly.

Target

| Metric                  |  Target |
| ----------------------- | ------: |
| Single Test             |  <50 ms |

---

# Common Anti-Patterns

Avoid:

* Testing multiple behaviors in one test
* Hidden dependencies
* Network calls
* Database access
* Sleeping or waiting
* Randomized inputs without control
* Assertions without clear expectations

# Pull Request Requirements

A pull request cannot be merged unless:

* All unit tests pass.
* Coverage thresholds are met.
* No flaky tests are introduced.
* New functionality includes corresponding unit tests.

---

# Best Practices

* Keep tests small and focused.
* Use reusable fixtures and factories.
* Prefer explicit assertions.
* Mock all external dependencies.
* Name tests to describe behavior.
* Refactor tests when production code changes.
* Treat failing tests as defects, not inconveniences.
* Review tests during code reviews with the same rigor as production code.

---

# Antigravity Development Instructions

When implementing unit tests:

1. Use **Jest** as the standard unit testing framework for both frontend and backend.
2. Follow the Arrange–Act–Assert structure consistently.
3. Mock all external dependencies, including databases, AI providers, queues, and third-party APIs.
4. Create reusable fixtures and mock factories for common domain objects.
5. Write descriptive test names that clearly explain the expected behavior.
6. Achieve or exceed the defined code coverage thresholds for every module.
7. Validate edge cases, error paths, and boundary conditions in addition to normal scenarios.
8. Ensure every new feature includes corresponding unit tests before merge approval.
9. Keep unit tests deterministic, isolated, and fast.
10. Integrate the unit test suite into the CI pipeline so failures block deployments.
