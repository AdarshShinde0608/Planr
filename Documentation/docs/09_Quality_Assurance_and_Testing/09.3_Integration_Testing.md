# Purpose

Integration testing verifies that independent modules and services communicate correctly.

Unlike unit tests, integration tests execute multiple real application components together to validate complete workflows.

The objective is to detect interface issues before End-to-End testing.

# Best Practices

* Test real service interactions whenever possible.
* Mock only third-party services.
* Keep test environments isolated.
* Reset the database before every test suite.
* Ensure repeatable, deterministic execution.
* Validate both success and failure paths.
* Include concurrency and transaction tests.
* Monitor execution time to keep the suite efficient.

---

# Antigravity Development Instructions

When implementing integration tests:

1. Create an isolated Docker-based integration testing environment using PostgreSQL, Redis, Backend API, AI Orchestrator, and BullMQ.
2. Execute real communication between internal services while mocking only external providers.
3. Validate complete workflows involving authentication, task creation, AI analysis, scheduling, notifications, and persistence.
4. Use Testcontainers (or equivalent) to provision temporary infrastructure during test execution.
5. Verify transactional consistency, rollback behavior, and concurrency handling.
6. Integrate the entire suite into GitHub Actions and block deployments on failure.
7. Maintain reusable test fixtures and datasets for different user personas.
8. Generate detailed reports containing execution metrics, failures, logs, and coverage.
9. Continuously monitor integration test execution time to avoid excessive CI duration.
10. Treat integration test failures as release-blocking issues until resolved.
